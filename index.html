A DevOps team manages multiple microservices, each with its own CodePipeline. All pipelines follow the same structure: Source → Build → Test → Deploy. The team wants to standardize pipeline creation and ensure consistent configuration across all services while allowing individual teams to customize deployment parameters. Which approach provides the BEST balance of standardization and flexibility?
  • Create a Lambda function that generates CodePipeline configurations based on input parameters. Store the function in a shared account and provide teams with an API Gateway endpoint to request new pipelines.
  • Develop an AWS CDK construct that encapsulates the standard pipeline structure. Publish the construct to AWS CodeArtifact as a reusable library. Teams import the construct and customize it by passing configuration parameters.
  • Create an AWS CloudFormation template with parameters for repository name, branch, and deployment configuration. Store the template in a central repository and use CloudFormation StackSets to deploy pipelines across multiple accounts and regions.
  • ✓ Use AWS Service Catalog to create a pipeline product with the standard structure. Define portfolio constraints and launch constraints. Teams provision pipelines through Service Catalog and provide customization parameters during launch.
  • Build a master CodePipeline template and manually clone it for each microservice. Document the customization process in a wiki and require teams to follow the documented steps.
Show Explanation
✓ Why correctAWS Service Catalog is purpose-built for this use case. It provides centralized governance, version control, and standardization while allowing controlled customization through parameters. Portfolio constraints ensure compliance, and the self-service model empowers teams while maintaining standards. Service Catalog also provides usage tracking and cost allocation.
✗ Why B is wrongWhile CDK constructs provide excellent code reusability, they require teams to have CDK expertise and maintain deployment infrastructure. Service Catalog provides a more accessible self-service interface and better governance controls without requiring infrastructure-as-code knowledge from consuming teams.
✗ Why A is wrongCloudFormation templates provide standardization, but StackSets are designed for multi-account/region deployment of identical resources, not for creating customized instances per microservice. This approach lacks the self-service and governance features that Service Catalog provides.
✗ Why C is wrongManual cloning is error-prone, doesn't scale, and provides no enforcement of standards. Documentation-based processes rely on human compliance and are difficult to audit. This approach lacks automation and governance controls.
✗ Why E is wrongBuilding a custom Lambda-based solution adds significant development and maintenance overhead. This approach requires custom authentication, authorization, error handling, and monitoring. It reinvents capabilities that Service Catalog provides out-of-the-box.
Quiz 2 — Sample Question
Domain 1: SDLC Automation (22%)
A DevOps engineer runs a custom test suite in AWS CodeBuild as part of a CodePipeline. The buildspec executes tests in the post_build phase using the command `npm test || true`. Despite test failures, the CodeBuild build reports as SUCCEEDED and the pipeline proceeds to the production deployment stage. The engineer needs test failures to fail the pipeline while still ensuring that test result artifacts are uploaded for debugging. Which solution correctly addresses this?
  • ✓ Remove `|| true` from the test command and move it to the build phase. Place artifact upload commands in the post_build phase, which executes regardless of the build phase outcome, ensuring test artifacts are always uploaded even when tests fail.
  • Keep `|| true` on the test command. Add a separate Lambda function triggered by CodeBuild completion events that parses CloudWatch Logs for test failure patterns and calls the CodePipeline PutJobFailureResult API if failures are found.
  • Remove `|| true` and run both the test command and artifact upload in the build phase using a shell script that captures the test exit code, uploads artifacts via the AWS CLI, and then exits with the original test exit code.
  • Configure CodePipeline to retry the test stage automatically on any failure with a retry count of 3. Add a CloudWatch alarm on CodeBuild failure metrics to notify the team if all retries fail.
Show Explanation
✓ Why A is correctRemoving `|| true` and moving the test command to the build phase ensures that a non-zero exit code from `npm test` causes CodeBuild to mark the build as FAILED, which stops the pipeline. The post_build phase executes regardless of whether the build phase succeeded or failed, making it the correct place for artifact upload and cleanup tasks. This leverages CodeBuild's built-in phase behavior without requiring custom logic.
✗ Why B is wrongKeeping `|| true` suppresses the exit code entirely, so CodeBuild always reports SUCCESS regardless of test outcomes. Adding a separate Lambda function to parse logs introduces unnecessary complexity, latency, and a race condition between the pipeline advancing and the Lambda checking results. The correct approach is to let CodeBuild's exit code mechanism work natively.
✗ Why C is wrongWrapping everything in a single build phase script that manually uploads artifacts before exiting is fragile. If the upload fails, the exit code from the upload command could mask the test exit code. CodeBuild's post_build phase is purpose-built for cleanup tasks and handles this separation cleanly without custom shell logic.
✗ Why D is wrongConfiguring CodePipeline to retry the entire stage on any failure does not distinguish between test failures and transient errors. Retrying test failures wastes time and does not address the root cause—the `|| true` suppressing the exit code. The pipeline would still eventually proceed if all retries also fail, and the fundamental exit code issue remains unresolved.

Domain 1: SDLC Automation (22%)
A company stores build artifacts in S3 with versioning enabled. The compliance team requires that artifacts must be retained for 7 years but frequently accessed artifacts (last 90 days) must be immediately available. Older artifacts can have retrieval delays. The solution must minimize storage costs. Which S3 configuration meets these requirements?
  • Use S3 Intelligent-Tiering for all artifacts. Configure archive configurations to move objects to Archive Access tier after 90 days and Deep Archive Access tier after 1 year.
  • ✓ Create an S3 Lifecycle policy that transitions objects to S3 Standard-IA after 90 days, to S3 Glacier Flexible Retrieval after 1 year, and to S3 Glacier Deep Archive after 3 years. Enable S3 Object Lock for compliance retention.
  • Use S3 Standard for all artifacts. Enable S3 Object Lock with retention mode set to 7 years. Accept higher storage costs for compliance requirements.
  • Store artifacts in S3 Standard for 90 days, then use Lambda to move them to S3 Glacier. After 7 years, use another Lambda function to delete artifacts. Use S3 versioning for audit trails.
Show Explanation
✓ Why correctS3 Lifecycle policies automate transitions between storage classes based on age. Standard-IA provides immediate access with lower costs for infrequent access. Glacier Flexible Retrieval and Deep Archive provide progressively lower costs for long-term retention with acceptable retrieval times. Object Lock ensures compliance retention for 7 years. This configuration optimizes costs while meeting access and compliance requirements.
✗ Why A is wrongWhile Intelligent-Tiering can work, it's designed for unpredictable access patterns and charges a monthly monitoring fee per object. The requirement has clearly predictable patterns (90 days frequent, then infrequent), so explicit lifecycle transitions (option B) provide more precise cost control without monitoring overhead. Intelligent-Tiering also doesn't include the explicit tiering to Glacier Deep Archive that option B provides for long-term cost savings.
✗ Why C is wrongUsing S3 Standard for 7 years is extremely expensive and doesn't meet the requirement to minimize costs. The requirement explicitly states older artifacts can have retrieval delays, indicating that cheaper storage classes should be used. This approach wastes significant money.
✗ Why D is wrongUsing Lambda functions to move objects is unnecessary when S3 Lifecycle policies provide this functionality natively. Lambda-based approaches add complexity, potential failure points, and operational overhead. S3 Lifecycle policies are more reliable and require no code maintenance.

Domain 1: SDLC Automation (22%)
A company deploys applications to Auto Scaling groups using CodeDeploy. During deployment, new instances are launched but the deployment fails with error "The deployment failed because the instance did not respond to the deployment lifecycle event within the timeout period." What is the MOST likely cause?
  • The deployment group's service role doesn't have permissions to communicate with Auto Scaling. Add autoscaling:* permissions to the service role.
  • ✓ The CodeDeploy agent is not installed on the AMI used by the Auto Scaling group launch template. New instances launch without the agent, preventing deployment.
  • The CodeDeploy agent is installed but not configured to start automatically on instance boot. The agent isn't running when deployment begins.
  • The Auto Scaling group's health check grace period is too short. Instances are terminated before the CodeDeploy agent can respond to lifecycle events.
Show Explanation
✓ Why B is correctFor Auto Scaling group deployments, the CodeDeploy agent must be pre-installed on the AMI. When new instances launch, they need the agent already installed and running to receive deployment instructions. If the AMI doesn't include the agent, new instances cannot participate in deployments, causing timeout errors. This is the most common cause of this error in Auto Scaling scenarios.
✗ Why A is wrongThe CodeDeploy service role needs Auto Scaling permissions for certain operations, but lack of these permissions would cause different errors (permission denied, unable to describe Auto Scaling groups). The timeout error indicates the agent isn't responding, which points to agent installation issues, not service role permissions.
✗ Why C is wrongWhile possible, most CodeDeploy agent installations configure automatic startup by default. If the agent were installed but not starting, this would be a configuration issue. However, the most common cause in Auto Scaling scenarios is the agent not being included in the AMI at all, making this a less likely root cause.
✗ Why D is wrongHealth check grace period affects when Auto Scaling starts health checks, not CodeDeploy agent responsiveness. If the grace period were too short, instances might be terminated by Auto Scaling, but the error message specifically mentions CodeDeploy lifecycle events, not health check failures. The issue is agent availability, not health check timing.

Domain 1: SDLC Automation (22%)
A company has three AWS accounts: development, staging, and production. The DevOps team wants a single CI/CD pipeline in a tools account that builds code once and deploys sequentially through staging and production. Artifacts must be accessible from all accounts. Which approach meets these requirements?
  • Create separate CodePipeline pipelines in each account. Use S3 event notifications to trigger the next account's pipeline after the previous one completes successfully.
  • ✓ Use a single CodePipeline in the tools account. Configure cross-account IAM roles in staging and production accounts that the pipeline assumes. Use a customer-managed KMS key to encrypt the artifact bucket and grant decrypt permissions to the target accounts.
  • Use a single CodePipeline in the tools account. Copy artifacts to S3 buckets in each target account using Lambda functions. Each account's CodeDeploy reads artifacts from its local S3 bucket.
  • Use AWS Organizations with a delegated administrator for CodePipeline. Organizations automatically handles cross-account deployment permissions and artifact sharing.
Show Explanation
✓ Why correctA single CodePipeline in a tools account with cross-account IAM roles is the AWS-recommended pattern for multi-account CI/CD. The pipeline's service role assumes a role in each target account to perform deployments. A customer-managed KMS key (not the default AWS-managed key) is required because the default key cannot be shared cross-account. The target accounts need both KMS decrypt permissions and S3 access to the artifact bucket.
✗ Why A is wrongSeparate pipelines per account do not provide a unified deployment workflow. Chaining pipelines via S3 events is fragile, hard to track end-to-end, and does not enforce deployment ordering or gating (such as requiring staging to pass before production).
✗ Why C is wrongCopying artifacts to each account's S3 bucket via Lambda adds unnecessary complexity. Cross-account roles allow CodeDeploy in target accounts to read directly from the pipeline's artifact bucket when the KMS key and bucket policy are properly configured. This extra copy step is not needed.
✗ Why D is wrongAWS Organizations does not automatically handle CodePipeline cross-account permissions. There is no delegated administrator concept for CodePipeline. Cross-account access must be explicitly configured with IAM roles, S3 bucket policies, and KMS key policies.

Domain 3: Resilient Cloud Solutions (15%)
A DevOps engineer is reviewing an application architecture for single points of failure. The application runs on a single EC2 instance in us-east-1a with an attached EBS volume and an Elastic IP address. Route 53 has an A record pointing to the Elastic IP. The backend uses a Single-AZ RDS PostgreSQL instance. Which TWO changes eliminate the most critical single points of failure? (Select TWO)
  • ✓ Replace the single EC2 instance with an Auto Scaling group spanning multiple Availability Zones behind an Application Load Balancer. Update Route 53 to alias the ALB DNS name.
  • ✓ Convert the RDS database from Single-AZ to a Multi-AZ deployment with a synchronous standby replica in another AZ and automatic failover.
  • Add a second Elastic IP address and configure Route 53 health checks to fail over between the two Elastic IPs attached to the same instance.
  • Enable EBS Multi-Attach to share the EBS volume across instances in multiple Availability Zones for redundancy.
  • Migrate the database from Amazon RDS to Amazon DynamoDB, which is inherently multi-AZ and requires no manual high-availability configuration.
  • Create daily AMIs of the EC2 instance and store them in S3. During failure, launch a replacement instance from the latest AMI in another Availability Zone.
Show Explanation
✓ Why A is correctThe single EC2 instance is the most critical compute-tier single point of failure. Placing instances in an Auto Scaling group across multiple AZs behind an ALB ensures that if one instance or an entire AZ fails, the ALB routes traffic only to healthy targets in surviving AZs. Auto Scaling replaces failed instances automatically, and Route 53 aliasing the ALB removes the dependency on a single Elastic IP.
✓ Why B is correctA Single-AZ RDS instance is a critical data-tier single point of failure. Multi-AZ creates a synchronous standby in another AZ; if the primary fails, RDS automatically promotes the standby (typically 60–120 seconds) and updates the DNS endpoint. This eliminates the database SPOF with no application code changes.
✗ Why C is wrongAdding a second Elastic IP does not help because both IPs would need to be associated with running instances to be reachable. With only one EC2 instance, the second EIP has nowhere to route traffic if that instance fails. The real fix is to eliminate the single-instance dependency (option A).
✗ Why D is wrongEBS Multi-Attach is available only for io1/io2 Provisioned IOPS volumes, works only within a single Availability Zone, and requires a cluster-aware file system. It cannot provide cross-AZ redundancy and does not address the compute SPOF.
✗ Why E is wrongWhile DynamoDB is inherently multi-AZ, migrating from a relational database to a NoSQL database is a significant re-architecture effort that changes data modeling, query patterns, and application code. It also does not address the compute-layer SPOF. Converting RDS to Multi-AZ (option B) solves the database SPOF with zero application changes.
✗ Why F is wrongDaily AMIs provide a restore mechanism, but they do not eliminate the single point of failure. If the instance fails, someone must detect the failure, launch a new instance from the AMI, and update DNS—resulting in extended downtime. This is a backup-and-restore approach, not a high-availability solution.

Domain 3: Resilient Cloud Solutions (15%)
A company runs a customer-facing web application on an EC2 Auto Scaling group behind an Application Load Balancer. Baseline traffic is 1,000 requests per second during business hours, but marketing campaigns cause gradual ramps to 5,000 requests per second over 30-minute periods. The team wants scaling that responds proportionally to load changes without manual intervention. Which Auto Scaling policy configuration meets these requirements with the LEAST operational overhead?
  • Configure a step scaling policy with CloudWatch alarms at CPU thresholds of 60%, 70%, and 80%, adding 2, 4, and 6 instances respectively at each threshold level
  • ✓ Configure a target tracking scaling policy that maintains the ALBRequestCountPerTarget metric at 1,000 requests per target with a 300-second instance warm-up period
  • Configure a scheduled scaling action to increase the desired capacity to 20 instances before each campaign and reduce capacity after the campaign ends
  • Configure a simple scaling policy with a single CloudWatch alarm at 70% CPU utilization, adding 5 instances per alarm breach with a 300-second cooldown period
Show Explanation
✓ Why B is correctTarget tracking scaling is the simplest and most proportional Auto Scaling policy. You specify a target value for ALBRequestCountPerTarget and Auto Scaling automatically adjusts the fleet to maintain that target — scaling out when requests per target exceed the value and scaling in when traffic drops. It handles both directions without manual threshold tuning. The 300-second warm-up prevents newly launched instances from being counted before they are ready to serve traffic, avoiding premature scale-in during ramp-ups. This provides the least operational overhead because AWS manages the scaling math, alarm creation, and adjustment logic automatically.
✗ Why A is wrongStep scaling policies require manually defining multiple CloudWatch alarms and corresponding scaling adjustments at each threshold. This increases operational overhead — you must tune thresholds, step sizes, and adjustment types. Step scaling is useful when you need non-linear scaling behavior, but for proportional response to variable load, target tracking is simpler and self-adjusting.
✗ Why C is wrongScheduled scaling requires advance knowledge of exactly when campaigns run. Marketing campaigns may not follow a fixed schedule, and traffic ramp duration varies. Scheduled scaling also doesn't adapt to actual load — if a campaign underperforms, you still pay for pre-provisioned capacity. It's appropriate for highly predictable patterns, not variable campaign timing.
✗ Why D is wrongSimple scaling uses a single alarm with a fixed adjustment and a cooldown period that blocks further scaling actions. During a 30-minute ramp to 5x traffic, the cooldown delays compound — the policy adds 5 instances, waits 300 seconds, re-evaluates, adds 5 more, waits again. This staircase pattern consistently under-provisions during ramps. Target tracking (option B) adjusts continuously without cooldown blocking.

Daily DataSync also gives an RPO of up to 24 hours, violating the sub-1-hour RPO requirement.
Quiz 30 — Sample Question
Mixed Review (Domains 1–6)
A global e-commerce company runs its application in us-east-1 and wants to expand to eu-west-1 for an active-active architecture. The application uses a relational database with strong consistency requirements for order processing, and the company needs sub-second failover for DNS. Which combination of services supports this active-active design?
  • DynamoDB global tables with Route 53 failover routing, CloudFront distribution, and Auto Scaling groups in each region.
  • RDS Multi-AZ deployment in us-east-1 with Route 53 failover routing to eu-west-1 and CloudFront for global caching.
  • ✓ Route 53 latency-based routing with health checks, Aurora Global Database with write forwarding, Auto Scaling groups in each region, and CloudFront.
  • Aurora read replicas in eu-west-1, Route 53 weighted routing, and Application Load Balancers in both regions.
Show Explanation
✓ Why correctRoute 53 latency-based routing directs users to the nearest region with sub-second DNS failover using health checks. Aurora Global Database provides cross-region replication with typically under 1-second lag and supports write forwarding, making it suitable for active-active with relational data. Auto Scaling in each region ensures capacity matches regional demand. CloudFront caches static content globally.
✗ Why A is wrongDynamoDB global tables provide multi-region replication but are a NoSQL service. The scenario specifies a relational database with strong consistency requirements for order processing, which is better served by Aurora Global Database. DynamoDB does not support SQL queries or relational data models needed for complex order processing.
✗ Why C is wrongAurora read replicas in the secondary region only handle read traffic — they cannot accept writes. This creates an active-passive architecture, not active-active. All write traffic would still go to us-east-1, creating a single point of failure for writes.
✗ Why D is wrongRDS Multi-AZ provides high availability within a single region by maintaining a synchronous standby replica in a different Availability Zone. It does not replicate data across regions, so it cannot support a multi-region active-active architecture.
